JavaScript, HTML & CSS
Sự khác biệt giữa var, let, const?
Bắt buộcDễ
  • var: function scope, hoisting (khai báo lên đầu hàm nhưng giá trị là undefined), có thể khai báo lại.
  • let: block scope, Temporal Dead Zone (không dùng trước khai báo), không khai báo lại trong cùng scope.
  • const: block scope, phải gán khi khai báo, không thể reassign — nhưng object/array vẫn mutable (nội dung thay được).

Quy tắc thực tế: mặc định dùng const, chỉ dùng let khi cần reassign, không dùng var.

Câu hỏi thêm hay gặp: "Temporal Dead Zone là gì?" → Là khoảng từ đầu block đến dòng khai báo let/const, truy cập biến trong vùng này ném ReferenceError.
this trong JavaScript là gì? bind / call / apply?
Bắt buộcKhó

this trỏ đến object đang gọi hàm — giá trị phụ thuộc vào cách gọi, không phải nơi định nghĩa (trừ arrow function).

  • Method call: obj.fn() → this = obj
  • Regular function call: fn() → this = undefined (strict) / global
  • Arrow function: không có this riêng, kế thừa từ scope bên ngoài
  • call(ctx, a, b): gọi hàm ngay với this = ctx
  • apply(ctx, [a, b]): như call nhưng args là array
  • bind(ctx): trả về hàm mới với this đã được fix cứng
const obj = { name: 'A', greet() { console.log(this.name); } }; const fn = obj.greet; fn(); // undefined (lost context) fn.call(obj); // 'A'
Event loop trong JavaScript hoạt động thế nào?
Bắt buộcKhó

JS là single-threaded. Cơ chế bất đồng bộ dựa trên Event Loop:

  • Call Stack: thực thi code đồng bộ.
  • Microtask Queue: Promise.then, queueMicrotask, MutationObserver — ưu tiên cao, chạy ngay khi stack trống.
  • Macrotask Queue: setTimeout, setInterval, I/O — chạy sau khi microtask queue rỗng.

Thứ tự: Sync → Microtasks (Promise) → Macrotasks (setTimeout).

console.log('1'); setTimeout(() => console.log('2'), 0); Promise.resolve().then(() => console.log('3')); console.log('4'); // Output: 1 → 4 → 3 → 2
Câu hỏi phổ biến nhất khi test JS nâng cao — nắm được output của đoạn code trên là ghi điểm lớn.
Promise và async/await là gì? Khác nhau thế nào?
Bắt buộcTrung bình

Promise: đối tượng đại diện giá trị sẽ có trong tương lai. 3 trạng thái: pending → fulfilled / rejected. Xử lý chuỗi bằng .then().catch().finally().

async/await: syntactic sugar trên Promise, code trông như đồng bộ, dùng try/catch để bắt lỗi.

// Promise chain fetch('/api/user').then(r => r.json()).then(data => ...).catch(err => ...); // async/await — dễ đọc hơn async function getUser() { try { const res = await fetch('/api/user'); const data = await res.json(); return data; } catch (err) { console.error(err); } }

Biết thêm: Promise.all() — chạy song song nhiều promise, Promise.allSettled() — không throw khi 1 cái fail.

Closure là gì? Cho ví dụ thực tế.
Bắt buộcKhó

Closure là hàm "nhớ" biến từ scope bên ngoài ngay cả sau khi scope đó đã kết thúc thực thi.

function makeCounter() { let count = 0; return { inc: () => ++count, get: () => count, }; } const c = makeCounter(); c.inc(); c.inc(); console.log(c.get()); // 2

Ứng dụng thực tế: debounce/throttle, module pattern, memoization, event handler trong loop, factory functions.

Prototype chain và kế thừa trong JavaScript?
Trung bình

Mọi object trong JS có [[Prototype]] trỏ đến object khác. Khi truy cập property không tồn tại trên object, JS leo lên chuỗi prototype cho đến Object.prototype rồi mới trả về undefined.

ES6 class là syntactic sugar của prototype-based inheritance. extends tạo chuỗi prototype, super() gọi constructor cha.

class Animal { constructor(name) { this.name = name; } speak() { return `${this.name} makes a noise.`; } } class Dog extends Animal { speak() { return `${this.name} barks.`; } }
ES6+ features quan trọng: destructuring, spread, optional chaining…
Bắt buộcDễ
  • Destructuring: const { name, age } = user / const [first, ...rest] = arr
  • Spread / Rest: {...obj, extra: true} — copy + override; ...args trong params
  • Optional chaining: user?.address?.city — không throw nếu null/undefined
  • Nullish coalescing: value ?? 'default' — chỉ fallback khi null/undefined (khác || vì 0 và "" không bị fallback)
  • Template literals: `Hello ${name}`
  • Arrow function: syntax ngắn + không có this riêng
  • Modules: import / export (ESM)
Debounce và Throttle là gì? Khi nào dùng?
Trung bình

Debounce: đợi sau lần gọi cuối một khoảng thời gian mới thực thi. Dùng cho search input — chỉ gọi API sau khi user ngừng gõ 300ms.

Throttle: đảm bảo hàm chỉ chạy tối đa 1 lần trong mỗi interval. Dùng cho scroll/resize event.

// Debounce đơn giản function debounce(fn, delay) { let timer; return (...args) => { clearTimeout(timer); timer = setTimeout(() => fn(...args), delay); }; }
Trong React: dùng useMemo hoặc thư viện lodash.debounce. Nhớ cleanup trong useEffect.
TypeScript cơ bản: type, interface, generics?
Trung bình

TypeScript = JavaScript + static typing. Giúp bắt lỗi lúc compile thay vì runtime.

  • type: alias cho type, có thể dùng union type ID = string | number
  • interface: mô tả hình dạng object, có thể extend, thường dùng cho props/API response
  • Generics: viết code tái sử dụng với type linh hoạt: function identity<T>(x: T): T
interface User { id: number; name: string; email?: string; } type ApiResponse<T> = { data: T; error: string | null }; function getUser(id: number): Promise<ApiResponse<User>> { ... }
Không cần TypeScript expert nhưng biết cơ bản là lợi thế lớn cho mức 7M. Nếu project React hiện tại của bạn dùng .tsx thì ghi điểm cao.
CSS Box Model và box-sizing?
Bắt buộcDễ

Box Model: content → padding → border → margin. Mặc định (content-box): width chỉ tính content, padding và border cộng thêm ra ngoài. Với box-sizing: border-box: width bao gồm cả padding + border — dễ tính toán layout hơn nhiều.

Hầu hết project dùng: * { box-sizing: border-box; } ở reset CSS.

Flexbox và Grid khác nhau thế nào? Dùng khi nào?
Bắt buộcTrung bình
  • Flexbox: 1 chiều (row hoặc column). Dùng cho navigation bar, button group, căn align items trong hàng/cột.
  • Grid: 2 chiều (rows + columns). Dùng cho page layout, dashboard, card gallery.

Thực tế: kết hợp cả hai — Grid cho macro layout trang, Flexbox cho micro-alignment bên trong component.

Biết thuộc tính quan trọng: Flex: flex-direction, justify-content, align-items, flex-wrap, gap. Grid: grid-template-columns, grid-template-rows, gap, grid-column, place-items.
Responsive design: media queries và các kỹ thuật?
Bắt buộcDễ

Mobile-first: viết CSS cho mobile trước, dùng min-width để mở rộng lên màn lớn hơn.

/* Mobile first */ .container { flex-direction: column; } @media (min-width: 768px) { .container { flex-direction: row; } }
  • Dùng rem / em thay px cứng cho font size
  • Dùng %, vw, vh, clamp() cho kích thước linh hoạt
  • Tailwind CSS: breakpoint prefix md:, lg: là utility class
  • Breakpoints phổ biến: 640 (sm), 768 (md), 1024 (lg), 1280 (xl)
JSON và XML là gì? Khi nào dùng cái nào?
Dễ

JSON: nhẹ, native trong JS (JSON.parse/stringify), phổ biến với REST API hiện đại. XML: verbose hơn, có namespace và schema validation, vẫn dùng trong SOAP, legacy systems, config (pom.xml, AndroidManifest.xml).

Web hiện đại: JSON là default. XML chủ yếu gặp khi làm việc với hệ thống cũ hoặc enterprise.

ReactJS — Hooks, Patterns & Performance
Virtual DOM là gì? React reconciliation hoạt động thế nào?
Bắt buộcTrung bình

Virtual DOM là bản sao nhẹ (JS object) của DOM thật, lưu trong memory. Khi state/props thay đổi:

  • React tạo VDOM mới
  • So sánh với VDOM cũ (diffing algorithm)
  • Chỉ cập nhật phần thực sự thay đổi vào DOM thật (reconciliation)

React Fiber (React 16+): engine reconciliation mới, chia nhỏ công việc để không block main thread, hỗ trợ concurrent features.

Lý do cần key trong list: giúp React nhận dạng item nào thay đổi/thêm/xóa thay vì re-render toàn bộ list.
useState và useEffect hoạt động thế nào?
Bắt buộcDễ

useState(init): trả về [state, setState]. Gọi setState sẽ schedule re-render. State update là bất đồng bộ — không đọc state mới ngay sau setState trong cùng function. Dùng functional update setState(prev => prev + 1) khi phụ thuộc vào giá trị trước.

useEffect(fn, deps): chạy side-effect sau render.

  • []: chạy 1 lần sau mount (giống componentDidMount)
  • [dep]: chạy khi dep thay đổi
  • Không có deps: chạy sau mỗi render
  • Return cleanup: chạy trước effect tiếp theo hoặc khi unmount
useEffect(() => { const controller = new AbortController(); fetch('/api/data', { signal: controller.signal }) .then(r => r.json()).then(setData); return () => controller.abort(); // cleanup }, [id]);
useRef hoạt động thế nào? Khi nào dùng?
Bắt buộcTrung bình

useRef() trả về object { current: value } tồn tại xuyên suốt lifecycle component. Thay đổi .current KHÔNG trigger re-render.

Dùng khi:

  • Truy cập DOM element trực tiếp: <input ref={inputRef} /> → inputRef.current.focus()
  • Lưu giá trị mutable không cần render lại: timer ID, previous value, flag
  • Tích hợp thư viện bên ngoài (D3, video player…)
function Timer() { const timerRef = useRef(null); const start = () => { timerRef.current = setInterval(..., 1000); }; const stop = () => clearInterval(timerRef.current); }
useReducer vs useState — khi nào dùng cái nào?
Trung bình

useState: đơn giản, state độc lập, ít logic.

useReducer(reducer, initialState): phù hợp khi:

  • State phức tạp (nhiều sub-values liên quan nhau)
  • Next state phụ thuộc vào state trước theo nhiều action
  • Muốn tách logic ra reducer function để dễ test
function reducer(state, action) { switch (action.type) { case 'increment': return { count: state.count + 1 }; case 'reset': return { count: 0 }; default: return state; } } const [state, dispatch] = useReducer(reducer, { count: 0 });
Kết hợp useReducer + Context API = lightweight Redux cho app vừa.
useMemo và useCallback khác nhau thế nào?
Bắt buộcKhó

useMemo(() => compute(), deps): memoize giá trị — chỉ tính lại khi deps thay đổi. Dùng cho tính toán nặng (sort, filter danh sách lớn).

useCallback(fn, deps): memoize function reference — trả về cùng reference khi deps không đổi. Dùng khi truyền callback xuống component con có React.memo.

Cạm bẫy: Đừng dùng bừa bãi — memoization có cost (so sánh deps). Chỉ dùng khi đo được vấn đề performance thực sự. Rule of thumb: dùng khi tính toán tốn > 1ms hoặc khi callback truyền vào memoized child.
Custom Hook là gì? Viết một custom hook đơn giản.
Bắt buộcTrung bình

Custom hook = function bắt đầu bằng use, gọi các hook khác bên trong, dùng để tái sử dụng logic stateful.

// useFetch — custom hook gọi API function useFetch(url) { const [data, setData] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { setLoading(true); fetch(url) .then(r => r.json()) .then(setData) .catch(setError) .finally(() => setLoading(false)); }, [url]); return { data, loading, error }; } // Sử dụng const { data, loading, error } = useFetch('/api/users');
Custom hooks hay gặp trong thực tế: useDebounce, useLocalStorage, useWindowSize, useIntersectionObserver, useForm. Biết giải thích và viết một cái là ghi điểm lớn ở mức 7M.
Props drilling là gì? Context API và State Management?
Bắt buộcTrung bình

Props drilling: truyền props qua nhiều tầng component trung gian không dùng đến. Giải pháp:

  • Context API: tốt cho state không thay đổi thường xuyên (theme, auth, locale). Dùng createContext + Provider + useContext.
  • Zustand: lightweight, ít boilerplate, phổ biến cho app vừa và nhỏ.
  • Redux Toolkit: cho app lớn, cần strict data flow, nhiều developer.
  • React Query / SWR: quản lý server state (cache, refetch, loading/error).
Biết phân biệt client state (UI state, form) vs server state (data từ API) — cách quản lý khác nhau. React Query giải quyết server state cực tốt.
Controlled vs Uncontrolled component?
Trung bình

Controlled: giá trị input quản lý bởi React state, cập nhật qua onChange. Toàn quyền kiểm soát — dễ validate, format real-time.

Uncontrolled: DOM tự quản lý, đọc giá trị qua ref. Đơn giản hơn cho trường hợp chỉ cần giá trị lúc submit.

Thực tế: controlled cho form có validation phức tạp; uncontrolled cho file input (<input type="file"> luôn uncontrolled) hoặc tích hợp thư viện non-React.

React Router hoạt động thế nào?
Bắt buộcDễ

Điều hướng client-side không reload trang. Cấu trúc cơ bản:

<BrowserRouter> <Routes> <Route path="/" element={<Home />} /> <Route path="/users/:id" element={<UserDetail />} /> <Route path="*" element={<NotFound />} /> </Routes> </BrowserRouter>

Hooks quan trọng: useNavigate() — navigate bằng code, useParams() — lấy URL params, useSearchParams() — query string, useLocation() — info về route hiện tại.

Làm sao tối ưu hiệu năng React app?
Khó
  • React.memo: bọc component con, tránh re-render khi props không đổi.
  • useMemo / useCallback: memoize giá trị và function.
  • Code splitting: React.lazy + Suspense — lazy load component khi cần.
  • Virtualization: react-window / react-virtual cho danh sách hàng nghìn item.
  • Tránh re-render không cần: đặt state gần nơi dùng, tách component nhỏ.
  • Image optimization: lazy loading, WebP, correct sizing.
  • Bundle size: tree shaking, dynamic import, phân tích với bundlephobia.
Dùng React DevTools Profiler để đo trước khi tối ưu — đừng tối ưu "mù". Câu trả lời có đề cập đến profiling là rất ấn tượng.
React patterns: HOC, Render Props, Composition?
Trung bình

HOC (Higher-Order Component): function nhận component, trả về component mới với logic bổ sung. Ví dụ: withAuth(ProtectedPage).

Render Props: truyền function vào prop để chia sẻ logic — ít dùng hơn kể từ khi có hooks.

Composition (children pattern): cách React-idiomatic nhất hiện nay.

// Compound component pattern <Select> <Select.Option value="a">Option A</Select.Option> <Select.Option value="b">Option B</Select.Option> </Select>
Với hooks, phần lớn use case của HOC và Render Props đã được thay bằng custom hooks. Biết pattern cũ vẫn cần vì nhiều thư viện cũ dùng HOC.
Vòng đời component React (lifecycle)?
Trung bình

Functional components dùng hooks mô phỏng lifecycle:

  • Mount: useEffect(() => { /* mount */ return () => { /* unmount */ }; }, [])
  • Update: useEffect(() => { /* dep changed */ }, [dep])
  • Unmount: cleanup function trong useEffect

Class component (biết để đọc code cũ): componentDidMount → componentDidUpdate → componentWillUnmount.

Strict Mode (React 18): mount → unmount → mount lại — để phát hiện side effect không clean. Nếu gặp effect chạy 2 lần khi dev, đây là lý do.
Error Boundary là gì?
Trung bình

Error Boundary là class component bắt JS error trong cây component con, hiển thị fallback UI thay vì crash toàn app. Dùng componentDidCatch và getDerivedStateFromError.

Chưa có hook tương đương — phải dùng class component hoặc thư viện react-error-boundary.

// Dùng thư viện react-error-boundary <ErrorBoundary fallback={<p>Something went wrong</p>}> <MyComponent /> </ErrorBoundary>
REST API, Authentication & Database
REST API là gì? Các HTTP method và status code?
Bắt buộcDễ

REST: kiến trúc API stateless, giao tiếp qua HTTP với resource-based URL.

  • GET: lấy dữ liệu (idempotent, không side effect)
  • POST: tạo mới resource
  • PUT: cập nhật toàn bộ resource (idempotent)
  • PATCH: cập nhật một phần
  • DELETE: xóa resource

Status codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized (chưa auth), 403 Forbidden (đã auth nhưng không có quyền), 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Internal Server Error.

Gọi API trong React: fetch vs axios? Error handling?
Bắt buộcDễ

fetch: native browser API, cần res.ok check thủ công (không throw với 4xx/5xx), cần res.json() thêm bước.

axios: tự parse JSON, throw với HTTP error status, support interceptor (tự thêm auth header, handle 401 global), timeout, cancel request.

// axios interceptor — tự thêm token axios.interceptors.request.use(config => { config.headers.Authorization = `Bearer ${getToken()}`; return config; }); // Global error handling axios.interceptors.response.use(null, error => { if (error.response?.status === 401) logout(); return Promise.reject(error); });
React Query / SWR: quản lý server state tốt hơn — auto cache, background refetch, loading/error state, pagination. Đề cập được là điểm cộng lớn.
JWT Authentication hoạt động thế nào?
Bắt buộcTrung bình

JWT (JSON Web Token): token gồm 3 phần header.payload.signature, encode bằng Base64. Server ký bằng secret key, client gửi lại mỗi request qua header.

  • Login → server trả JWT (access token + refresh token)
  • Client lưu vào localStorage hoặc httpOnly cookie
  • Mỗi request: Authorization: Bearer <token>
  • Server verify signature, decode payload lấy user info
  • Access token hết hạn → dùng refresh token lấy token mới
Bảo mật: Lưu token trong httpOnly cookie an toàn hơn localStorage (tránh XSS). Biết điều này là điểm cộng với employer.
CORS là gì? Xử lý thế nào?
Trung bình

CORS (Cross-Origin Resource Sharing): browser chặn request từ origin khác (domain, port, protocol khác) để bảo mật (Same-Origin Policy).

Server cần trả header Access-Control-Allow-Origin: * (hoặc origin cụ thể). Preflight request (OPTIONS) được gửi trước với method phức tạp.

Khi dev: dùng proxy trong vite.config.js:

server: { proxy: { '/api': { target: 'http://localhost:3001', changeOrigin: true } } }
SQL cơ bản: JOIN, GROUP BY, subquery?
Bắt buộcTrung bình
  • INNER JOIN: rows khớp ở cả 2 bảng
  • LEFT JOIN: tất cả rows bảng trái + khớp bên phải (NULL nếu không có)
  • GROUP BY: gom nhóm để dùng aggregate (COUNT, SUM, AVG, MAX, MIN)
  • HAVING: lọc sau GROUP BY (WHERE không dùng được với aggregate)
  • ORDER BY: sắp xếp kết quả
  • LIMIT / OFFSET: phân trang
-- Lấy users có tổng đơn hàng > 5 SELECT u.name, COUNT(o.id) AS order_count FROM users u INNER JOIN orders o ON u.id = o.user_id GROUP BY u.id, u.name HAVING COUNT(o.id) > 5 ORDER BY order_count DESC;
SQL vs NoSQL (MySQL/PostgreSQL vs MongoDB)?
Trung bình
  • SQL (MySQL, PostgreSQL): schema cố định, ACID transactions, quan hệ rõ ràng. Phù hợp: tài chính, ERP, app có quan hệ phức tạp. PostgreSQL mạnh hơn MySQL (JSON support, full-text search, extensions).
  • MongoDB (NoSQL): document-based, schema linh hoạt, scale horizontal tốt, query JSON native. Phù hợp: catalog, CMS, real-time data, dữ liệu cấu trúc biến đổi.
Netviet yêu cầu MySQL/PostgreSQL chính, MongoDB là lợi thế. Ưu tiên ôn MySQL — biết CRUD, JOIN cơ bản là đủ cho intern.
Database indexing là gì? Khi nào cần dùng?
Trung bình

Index: cấu trúc dữ liệu (B-tree) giúp DB tìm kiếm nhanh hơn thay vì quét toàn bảng (full table scan).

  • Nên index: cột thường dùng trong WHERE, JOIN, ORDER BY; foreign keys.
  • Không nên index mọi cột: index tốn storage và làm chậm INSERT/UPDATE/DELETE.
  • EXPLAIN / EXPLAIN ANALYZE: xem query plan để phát hiện chỗ cần index.
Biết khái niệm này tuy là DB kiến thức cho fresher nhưng sẽ tạo ấn tượng tốt — cho thấy bạn không chỉ biết "CRUD" mà còn nghĩ đến performance.
Error handling pattern trong React app?
Trung bình
  • API errors: try/catch với async/await, lưu error state, hiển thị toast / error message.
  • Network error vs HTTP error: phân biệt fetch fail (network) vs 4xx/5xx (server).
  • Global handler: axios interceptor bắt 401 (redirect login), 500 (hiện thông báo chung).
  • Error Boundary: bắt render error không ngờ tới.
  • Form validation: validate ở client trước khi gửi API.
async function submitForm(data) { setLoading(true); try { await api.post('/submit', data); toast.success('Thành công!'); } catch (err) { const msg = err.response?.data?.message ?? 'Có lỗi xảy ra'; toast.error(msg); } finally { setLoading(false); } }
Git & DevOps cơ bản
Các lệnh Git thường dùng hàng ngày?
Bắt buộcDễ
git clone <url> # clone repo git status # xem trạng thái git add . / git add <file> # staging git commit -m "feat: add login" # commit git push origin <branch> # đẩy lên remote git pull # lấy code mới nhất git checkout -b feature/x # tạo branch mới git merge <branch> # merge git stash / git stash pop # tạm lưu changes git log --oneline # xem history git diff # xem thay đổi chưa stage
git reset vs git revert — khác nhau thế nào?
Trung bình

git reset: di chuyển HEAD về commit trước — viết lại history. Chỉ dùng trên branch cá nhân chưa push.

  • --soft: giữ changes ở staging
  • --mixed (default): giữ changes ở working dir
  • --hard: xóa hoàn toàn changes ⚠️

git revert <hash>: tạo commit mới đảo ngược commit đó — an toàn cho shared branch, không viết lại history.

Quy tắc vàng: Không bao giờ reset/force push branch đã có người khác pull. Dùng revert cho main/develop.
Merge vs Rebase khác nhau thế nào?
Trung bình

Merge: tạo merge commit mới, giữ nguyên history — an toàn, thể hiện đúng lịch sử làm việc song song. Dùng cho shared branch.

Rebase: "di chuyển" commits của branch lên đầu nhánh đích, history tuyến tính và sạch hơn. Dùng cho branch cá nhân trước khi tạo PR.

# Rebase feature branch lên develop mới nhất git checkout feature/x git rebase develop # Sau đó merge vào develop (sẽ là fast-forward)
Xử lý merge conflict thế nào?
Bắt buộcTrung bình

Conflict xảy ra khi 2 branch sửa cùng dòng code.

  • Git đánh dấu conflict bằng <<<<<<<, =======, >>>>>>>
  • Mở file, chọn giữ code nào (hoặc kết hợp cả hai)
  • git add <file> để đánh dấu resolved
  • git commit hoàn tất merge

Dùng VSCode / GitLens để visualize conflict dễ hơn. Trao đổi với người viết code kia nếu không chắc nên giữ phần nào.

Phòng tránh conflict: Pull thường xuyên, branch nhỏ và ngắn, không để branch tồn tại quá lâu mà không sync.
Git workflow trong team (feature branch, PR, commit message)?
Bắt buộcTrung bình

Quy trình thường gặp: main (production) ← develop (staging) ← feature/xxx (cá nhân).

  • Tạo branch từ develop: git checkout -b feature/login-page
  • Commit nhỏ, thường xuyên với message rõ ràng
  • Push → tạo Pull Request → code review → merge
  • Không push thẳng vào main/develop

Conventional Commits: feat: add login form, fix: correct email validation, chore: update deps, refactor:, docs:, style:.

Biết gì về CI/CD, Docker, Cloud (AWS/Azure/GCP)?
Lợi thế

Đây là điểm cộng — không bắt buộc với intern, nhưng biết cơ bản thì ấn tượng:

  • CI/CD: tự động build, test, deploy khi push code (GitHub Actions, GitLab CI). Ví dụ: push lên develop → tự chạy test → deploy staging.
  • Docker: đóng gói app + dependencies vào container, chạy nhất quán mọi môi trường. Lệnh cơ bản: docker build, docker run, docker-compose up.
  • Cloud: AWS S3 + CloudFront (host static site), EC2 (server), Vercel/Netlify (deploy frontend đơn giản nhất).

Nếu chưa biết: "Em chưa có kinh nghiệm thực tế nhưng đã deploy frontend project lên Vercel và đang tìm hiểu Docker cơ bản."
